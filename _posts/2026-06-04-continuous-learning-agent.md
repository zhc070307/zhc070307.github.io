最近我连续读了几篇关于持续学习、Agent 记忆和模型适应的论文。它们讨论的对象放在一起看会形成一条很有意思的线索：如果我们希望 AI Agent 不只是一次性完成任务，而是在长期使用中不断和用户磨合、积累经验、修正行为，那么它到底应该怎样“学习”？我最开始关注的是 XSkill: Continual Learning from Experience and Skills in Multimodal Agents。这篇论文代表了现在agent记忆一个比较乐观的方向：不更新模型参数，而是让 Agent 从过去轨迹中提炼经验和技能，存到外部知识库中，之后在新任务里检索、适配和调用。后来我又读到 Useful Memories Become Faulty When Continuously Updated by LLMs，这篇文章几乎是给前一种路线泼了一盆必要的冷水：当前 LLM 并不是可靠的记忆巩固器，持续把经验抽象成记忆，可能使记忆越来越坏，甚至低于没有记忆的基线。中间还有一篇看似不直接属于 Agent memory 方向、但对我启发很大的论文：Learning, Fast and Slow: Towards LLMs That Adapt Continually。这篇论文主要讨论 LLM 后训练阶段的快慢学习：模型参数是慢权重，prompt/context 是快权重，二者应该在训练过程中共同优化。它和 XSkill、Faulty Memory 的研究对象不完全一样，但它给了我一个很重要的概念框架：学习不一定只发生在模型权重里，也可以发生在模型外围的上下文、记忆、工具配置和运行框架中。

这篇博客想整理的就是这一条思路：从 XSkill 的外部经验—技能记忆，到 Faulty Memory 对持续记忆更新的警示，再到 FST 给出的快慢学习框架，最后提出一个我自己的想法：未来的个人 Agent 可能需要一个快慢双系统。快系统是不断更新的 Agent harness，慢系统则可能是端侧小模型或个性化适配层，而云端大模型可以作为教师、审计者和复杂推理器参与其中。

## 一、为什么 Agent 需要持续学习

和人类的学习方式相比，现在很多 AI agent有一个缺陷：它们很聪明，但不够定制化。它们可以在单次任务中表现得很好，可以读文档、写代码、调用工具、回答复杂问题，但任务结束后，它们经常不会真正吸收这次经验，作出有效的改进。下次遇到相似问题时，它还是像第一次一样重新判断、重新探索、重新犯错。这和人类的学习方式很不一样。一个人和某个用户、某个工作流、某类任务相处久了，会逐渐形成习惯和方法。例如，一个研究助理反复帮你读论文后，会知道你喜欢先看问题意识，再看方法框架，再看实验和局限；又比如，一个长期合作的程序员会知道你偏好怎样的项目结构、怎样的解释粒度、怎样的代码风格。这些东西不是一次任务里临时生成的，而是在长期互动中积累出来的。如果我们，长远来看，真的想要一个个人 Agent，那么它不能只是一个无状态的工具调用器。它应该能从过去经历中学习，知道用户的偏好、项目的上下文、过去哪些方法有效、哪些错误应该避免。问题是，这种学习应该发生在哪里？

最直接的想法是更新模型参数。但它的代价很昂贵，也很危险。因为参数更新难以审计、难以回滚，而且可能造成遗忘。另一条路线是更新模型外部的系统—Harness—：记忆、技能、prompt、工具策略、工作流、检索器。这就是 Agent 的外部记忆路线，也是 XSkill 这类论文讨论的重点。

## 二、XSkill：一个外部经验—技能记忆系统

XSkill 讨论的是多模态工具 Agent。这里的多模态工具 Agent 不只是能看图和回答问题，还可以调用代码工具处理图像，使用网页搜索或图像搜索获取信息。它面对的不是单纯语言推理，而是需要在视觉观察、工具调用和多步决策之间不断切换。这类 Agent 的问题往往不是“没有工具”，而是“不会稳定地使用工具”。例如图片倒置时，它可能没有先旋转；目标太小时，它可能没有先裁剪放大；识别未知物体时，它可能没有先使用图像搜索；搜索失败后，它可能不知道换一种方式恢复。换句话说，能力是有的，但行动策略不够稳定，工具编排不够灵活。

XSkill 的**核心想法**是：Agent 应该从过去轨迹中积累两类外部知识，一类叫经验，一类叫技能。

* 经验是动作级的、局部的、情境敏感的知识。它通常回答的是：在某种具体情境下，下一步应该怎么做。例如，当图像太暗时，先增强亮度；当目标太小时，先裁剪和放大；当无法识别未知物体时，考虑使用图像搜索。经验不是完整流程，而是一种局部判断，它让 Agent 在执行过程中更少走弯路。
* 技能是任务级的、结构化的、可复用的工作流。它回答的是：某类任务通常应该怎样组织步骤和工具。例如，识别图片中未知对象时，可以先定位目标区域，再裁剪目标，再进行图像搜索，最后访问网页并综合回答。技能更像一份操作手册，帮助 Agent 进行高层规划和工具编排。

这个“经验—技能”的双流设计很重要。因为很多 Agent memory 方法容易把所有内容都简单叫做 memory，但 XSkill 进一步区分了不同粒度的知识。经验负责临场反应，技能负责整体流程；经验让 Agent 更灵活，技能让 Agent 更稳定。对于一个长期运行的 Agent 来说，这种分层比单纯堆积记忆更合理。

XSkill 的第二个关键点是“视觉锚定”。在多模态任务中，很多关键失败原因不在文字日志里，而在图像状态里。比如，文字轨迹只会告诉我们“搜索失败”，但真正原因可能是图片倒置、目标太小、光线太暗，或者没有先裁剪关键区域。如果只根据文字轨迹总结经验，Agent 很难学到真正有用的视觉行动策略。因此，XSkill 在抽取经验和技能时，会把图像观察、工具调用、中间结果和任务成败结合起来。它不是只总结“调用了什么工具”，而是试图总结“什么视觉状态触发了什么工具动作，以及这个动作为什么有用”。这也是它相比普通文本型 Agent memory 更有针对性的地方。

第三个重要设计是“检索后适配”。XSkill 不是把旧经验原封不动写成 prompt，而是先根据当前任务检索相关经验和技能，再根据当前图像和问题把它们重写成具体建议。比如旧经验可能只是说“目标太小时先裁剪”，但当前任务中它会被改写成“先定位图片角落中的两个小目标，对它们进行裁剪和放大，再使用图像搜索或网页搜索进行识别”。旧知识只有经过这种适配，才不会变成僵硬模板。

XSkill 还有一个我认为很值得注意的设计，那就是执行模型和知识管理模型分工。执行模型负责实际做任务，知识管理模型负责总结轨迹、提取经验、生成技能、合并知识、检索知识和适配知识。论文还提到，知识管理可以使用更强的模型，而执行模型可以保持灵活。这一点很有启发，因为它说明 Agent 的学习系统不必完全压在同一个模型身上。一个模型可以负责行动，另一个更强的模型可以负责反思和知识整理。

所以，XSkill 的范式可以概括为：不更新模型参数，而是通过外部经验—技能知识库，让 Agent 在任务之间积累程序性知识。这里的程序性知识不是事实知识，而是“怎么做事”的知识。它不是普通 RAG 检索某篇文档说了什么，而是检索过去任务中总结出来的工具使用方法、错误恢复策略和工作流。这个方向非常吸引人。它提供了一种无需参数更新的持续学习路径：Agent 可以通过改变自己的外部知识结构，在不改变模型本体的情况下变得更熟练。

## 三、Faulty Memory：持续更新的记忆可能越改越坏

我们可能会对外部记忆系统很乐观，但 Useful Memories Become Faulty When Continuously Updated by LLMs 这篇文章提醒我们：由 LLM 自己持续总结和更新的记忆很可能出现系统性问题。这篇论文的核心问题是：如果 Agent 不断把过去轨迹压缩成文本记忆，并且每次新交互后都更新这套记忆库，记忆真的会越来越好吗？作者的回答是否定的。记忆效用可能一开始上升，但随着持续巩固和重写，会逐渐下降，甚至低于没有记忆的基线。

这篇论文最重要的概念区分是情景记忆和抽象记忆。情景记忆是原始经历本身，包括任务、观察、行动步骤、工具调用、错误、反馈和结果。它像原始案卷，信息量大，不够压缩，但保留了证据。抽象记忆则是从多个原始经历中总结出的 lesson、规则、技能或经验。它更短、更可复用，但需要 LLM 进行抽象和重写。过去很多 Agent memory 系统追求抽象记忆，因为长期运行的 Agent 不可能无限保存所有轨迹。但这篇论文指出：抽象本身是有损的。每一次总结都可能丢掉细节、删掉适用条件、引入错误规则，或者把不该合并的经验合并到一起。

文章给出了一个特别有力的强反例：模型在没有记忆的情况下本来可以解决一组 ARC-AGI 问题，而且记忆构建时还提供了真实解。也就是说，输入给记忆系统的经验不是失败轨迹，也不是噪声，而是高质量、正确的经验。按理说，基于真实解总结出的记忆至少不应该伤害模型。但实验显示，经过流式巩固后，模型反而在原来会做的问题上失败了。这个结果说明，问题不在于经验本身无用，而在于巩固过程把有用经验变成了错误记忆。

论文还发现，同一批经验，用不同更新方式会产生不同质量的记忆。一次性看完整个轨迹池再总结，和流式地一批批更新，是不一样的；按任务类型分组总结，和把不同任务混在一起总结，也是不一样的。流式更新更容易导致记忆侵蚀，异质任务混在同一批中更容易让模型总结出错误规则。这点对真实 Agent 特别重要。真实 Agent 在部署中遇到的就是流式环境：任务一个接一个来，用户偏好逐渐变化，工具结果不断产生。因此它不可能总是离线地、完整地、按任务族整理好全部历史再总结。也就是说，真实 Agent 正处在这篇论文认为最容易产生 faulty memory 的场景中。

这篇论文还做了一个重要对照：直接保留原始轨迹不做抽象，有时已经足够强。把原始轨迹作为上下文示例提供给模型，效果常常可以和抽象记忆方法竞争，甚至超过它们。原因很直观：原始轨迹保留了观察、行动、失败、反馈和具体情境，模型可以通过上下文学习从具体案例中获得帮助。如果抽象过程不可靠，直接保留证据反而是更好的办法。

作者总结了三类错误记忆机制。

* 第一是错误分组。模型在抽象前需要判断哪些经验属于同一类，哪些可以合并。如果它把底层结构不同的任务合并到一起，就会总结出错误规则。
* 第二是过度泛化。原本只适用于某种情境的经验，被模型写成了普遍规则，结果在相邻但不同的任务中误导 Agent。
* 第三是窄样本过拟合。如果输入流里有大量相似例子，模型可能把这些例子的偶然特征当成一般规律，导致记忆在新例子上泛化不好。

这篇论文的结论不是“Agent 不应该有记忆”，而是：Agent 的记忆系统不能只是一个不断重写的抽象库。原始情景应该被作为一等证据保留，抽象应该被门控。也就是说，系统不应该每次交互后都自动总结，而应该判断是否真的有足够证据进行抽象，哪些细节必须保留，哪些记忆需要删除，哪些抽象需要回滚。这篇论文的价值在于，它把 Agent memory 的研究从“如何记住更多”推进到了“如何可靠地记住”。记忆系统不仅要会写入，更要妥善的治理。

## 四、FST：为什么快慢学习的框架仍然重要

虽然 Learning, Fast and Slow 这篇论文和上面两篇不是同一个直接问题——它主要讨论 LLM 后训练阶段，而不是部署态 Agent memory。但它对我的思考非常关键，因为它提供了一个更一般的学习框架：学习不应该只发生在一个通道里。FST 的核心是把大语言模型的适应分成两类：慢权重和快权重。慢权重是模型参数，通过 RL 等方式更新，成本高、影响持久，适合承载长期能力。快权重是 prompt、context 或 textual scaffold，也就是文本上下文。它们不是真正的神经网络参数，但会显著影响模型行为，而且可以快速修改。

这篇论文的主张是：LLM 后训练不应该只更新参数，也应该同时优化上下文。传统 RL 后训练通常固定 prompt，然后把所有任务适应都写进参数里，这可能导致模型偏离基座模型、遗忘原有能力，并降低未来学习新任务的可塑性。FST 则让 prompt/context 和模型参数共同演化：一边用 GEPA 这样的提示优化方法更新上下文，一边在这些上下文条件下用 RL 更新模型参数。

这篇论文的“continual”主要发生在训练过程中。也就是说，它不是说模型部署后会在普通使用中自动学习，而是说在连续后训练任务流中，模型面对不断变化的任务时还能继续适应。它的价值在于指出：如果所有学习压力都压到参数上，模型可能变得过度专门化；如果让上下文承担一部分任务适应，参数就不需要移动那么远，模型也更可能保持可塑性。这和 Agent memory 的关系不是直接的，但给了我一个迁移思路：**如果在 LLM 后训练中，prompt/context 可以作为快学习通道，那么在 Agent 系统中，整个 harness 都可以被视为快学习通道**。这里的 harness 指的是包裹在模型外面的运行系统，包括 memory、skills、prompt、工具配置、检索器、工作流、反思规则、评价器、用户偏好和任务模板。对于 Agent 来说，快权重不应该只是一段 prompt，而可以是整个可变的外部运行结构。

于是，我们可以把 FST 的思想从“模型训练”迁移到“Agent 学习”上：模型参数是慢系统，Agent harness 是快系统。快系统通过记忆和技能快速适应当前用户和任务，慢系统则通过更低频、更谨慎的方式吸收长期稳定的能力或偏好。

这个迁移也让我想到另一个问题：能不能把模型的 reinforcement learning 思路迁移到 Agent 的 reinforcement learning？在普通 LLM RL 中，模型生成答案，系统根据 reward 更新参数。而在 Agentic RL 中，Agent 不是只生成一个答案，而是在环境中连续行动：观察、调用工具、获得反馈、再行动，最后完成任务。这里的学习对象不一定只是模型参数，也可以是 Agent 的工具策略、记忆更新规则、技能库、任务分解器和工作流。也就是说，Agent 的强化学习可以不只训练模型本体，还可以训练或改进它的 harness。这就是为什么 FST 虽然不是 Agent memory 论文，但仍然应该放进这条思路里。它提供了“快系统与慢系统分工”的概念基础，而我们可以进一步把这个框架推广到 Agent：快学习发生在 harness，慢学习发生在模型或端侧个性化层。

## 五、我看到的核心矛盾：记忆需要压缩，但压缩会出错

把这几篇文章放在一起看，我觉得最核心的矛盾是：Agent 必须从经验中学习，否则它永远只是一次性工具调用器；但经验又不能无限保存，必须压缩、抽象和整理；而当前 LLM 又不可靠，抽象过程可能把有用经验变成错误记忆。这就是 continuous agent memory 的根本困难：如果我们只保留原始轨迹，信息最完整，也最可追溯，但它会无限增长。上下文长度有限，检索成本会上升，隐私和存储也会成为问题。一个长期使用的个人 Agent 不可能永远保留所有细节并在每次任务中全部考虑。如果我们把轨迹压缩成抽象记忆，系统会更轻量，更容易迁移，也更像真正的经验学习。但第三篇论文告诉我们，抽象不是免费的。错误分组、过度泛化、适用条件丢失、流式更新侵蚀，都可能让记忆越来越差。

所以，未来的 Agent memory 不能只是在“保留原始轨迹”和“总结抽象经验”之间二选一。我目前的一个想法是引入分层。

* 第一层应该是原始情景证据。它不一定全部永久保留，但至少在记忆被抽象之前和之后，系统应该能够追溯抽象来自哪些原始经验。
* 第二层可以是经过门控的抽象记忆。不是每次交互都自动总结，而是在有足够相似经验、适用条件清楚、经过验证后，再形成经验、技能或工作流。
* 第三层可能是更慢的个性化模型。那些长期稳定、反复验证的用户偏好和任务模式，或许不应该永远停留在外部 memory 中，而可以周期性进入端侧小模型或本地 adapter。

这样一来，Agent 的学习就不只是“记忆库变大”，而是一个从原始经验到抽象技能，再到长期个性化内化的过程。

## 六、我的设想：个人 Agent 的快慢双系统

基于这些论文，我现在有一个初步设想：未来个人 Agent 的持续学习可能需要一个快慢双系统。快系统是 Agent harness。它包括原始情景记忆、抽象经验、技能库、用户偏好、工具策略、检索器、workflow、prompt 和反思规则。它的特点是更新快、成本低、可编辑、可回滚。用户今天指出一个偏好，Agent 明天就应该能使用；某个任务失败后，Agent 也应该能立刻记录错误和恢复策略。但快系统不能无限膨胀。第三篇论文提醒我们，记忆会变脏，抽象会变坏。快系统适合短期适应和可审计记忆，但不适合承载所有长期个性化。慢系统可以是端侧小模型，或者本地个性化 adapter。它不是要替代云端大模型，而是负责吸收长期稳定、反复验证的用户模式。例如，用户长期偏好的解释风格、写作语气、项目组织方式、常用工具链、论文阅读习惯、代码风格等。这些东西经过足够多次验证后，可以从 harness 迁移到端侧模型中。这里的关键是“临界点”。系统不应该看到一条记忆就训练模型，而应该等到某些模式反复出现，并且经过验证后，才启动慢学习。这个过程可以像这样：用户和 Agent 长期交互，快系统先保留原始情景、用户反馈、任务结果和临时经验。经过一段时间后，系统检查哪些偏好稳定，哪些工作流反复有效，哪些经验被多次验证。然后，云端大模型参与审计和整理，把这些稳定模式转化为训练样本、偏好数据、反例和小型评估集。最后，端侧小模型或 adapter 进行低频更新，并通过验证集检查是否真的改善了用户体验。如果更新失败，就回滚。

这种架构可以避免两个极端。一个极端是永远只堆 memory，最后记忆越来越大、越来越脏；另一个极端是频繁更新模型参数，把错误经验写进模型里，变得难以审计和回滚。更合理的路线是：短期靠 harness 快速适应，长期靠端侧模型低频内化。

## 七、云端大模型与端侧小模型的协同

我觉得另一个关键点是，不应该把端侧小模型理解成云端大模型的缩小替代品。把一个强大的通用大模型完整压缩到端侧，并且还希望能力不损失，目前来看这是技术上非常困难的。所以更合理的方向是大小模型协同。云端大模型负责复杂推理、高质量记忆审查、训练数据生成和困难任务。端侧小模型负责本地个性化、隐私保护、用户偏好建模、本地检索和上下文组装。这样，小模型不需要成为全能模型，它只需要成为“最懂这个用户的模型”。例如，端侧小模型可以先判断当前任务是否涉及隐私，哪些本地记忆可以使用，哪些信息不应该发给云端。它可以把用户偏好和本地上下文压缩成更好的 prompt，发送给云端大模型。云端大模型完成复杂推理后，端侧小模型再根据用户风格进行最后调整。更进一步，云端大模型还可以周期性帮助端侧小模型生成训练数据和评估集，使端侧小模型逐渐变得更适合这个用户。

这样，一个个人 Agent 的架构可能是：云端大模型提供通用智能和复杂推理能力；端侧小模型提供长期个性化和隐私边界；Agent harness 提供短期记忆、工具调用和任务工作流；三者共同构成一个持续学习系统。我觉得这比单纯讨论“模型要不要更新参数”更接近真实产品的形态。因为真正的个人 Agent 既需要强大的通用能力，又需要本地的长期个性化；既需要快速记忆，也需要慢速内化；既需要灵活，又需要可审计和可回滚。

## 八、一个可能的研究问题

把上面的想法整理成研究问题，我觉得可以这样表述：如何构建一个面向个人 Agent 的快慢持续学习系统，使其能够通过外部 harness 快速适应用户，同时将长期稳定、经过验证的用户模式低频迁移到端侧小模型中，并利用云端大模型进行记忆审计、数据蒸馏和复杂推理？

这个问题包含几个具体挑战。

* 第一，记忆治理。系统如何判断哪些原始轨迹应该保留，哪些可以删除，哪些可以抽象？如何避免第三篇论文指出的错误分组、过度泛化和记忆侵蚀？
* 第二，抽象门控。什么时候应该把经验变成技能？什么时候应该保留原始证据？什么时候应该拒绝总结，因为证据不足或任务结构不清楚？
* 第三，快慢迁移。哪些内容适合留在 memory 或 skill 里，哪些内容适合进入端侧模型？这个边界很重要。临时偏好不应该进入参数，长期稳定模式才值得慢学习。
* 第四，云端—端侧协同。云端大模型如何帮助端侧小模型学习，而不直接暴露全部隐私数据？端侧模型如何筛选、压缩和脱敏本地经验？云端模型生成的训练数据如何被审计？
* 第五，评估与回滚。端侧模型更新之后，如何判断它真的更懂用户，而不是过拟合错误偏好？如果更新后表现变差，如何回滚？如何保留外部证据来解释模型为什么改变？

这些问题连接了三条研究线：Agent memory、Agentic RL / harness optimization、端侧个性化模型。

Agent 如何在长期使用中，带着证据、带着审计、带着可回滚机制，逐步成为一个更懂用户的系统？
